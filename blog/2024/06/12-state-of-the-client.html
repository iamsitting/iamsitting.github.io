The State of the Client

  • June 12, 2024

This is a post on frontend web architecture. Why? Because it's the hardest slice of software do right.

I have been pondering this topic for several months and have been studying it deeply for the last couple of weeks. Now I want to take the opportunity to talk about it and reflect on my own experiences.

The Web Client #

Web development is a pretty fascinating form of software develoment because to fully operate it requires two machines: a server and a client. Yes, many applications leverage a server-client model, but only in web applications are a server and a client a necessity.

Consider a mobile application or a desktop application. You download an app (often a packaged binary), you install it and then it runs. Though it may leverage some remote services, fundamentally, the application data, state, and user interface all live in the same machine.

In a web application, data and rules (such as roles and permissions) live on the web server and the UI lives on the web client. And here is where we run into a problem. Because the client is responsible for the UI, the client must manage a state. Or does it?

HTTP Architecture#

Now, let's talk about HTTP architecture. HTTP, in simplest, terms is just a request and response protocol. The client makes the request and the server sends a response. Traditionally, the server served files and was responsible in ensuring that the client had acccess to those files.
